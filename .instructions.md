# Twin-Stick Tower Defense Game – Project Architecture

## Game Overview

**Genre:** Twin-stick shooter tower defense with flow field pathfinding.

**Core Concept:**
- Player character moves and aims independently (twin-stick controls).
- Player can place defensive structures (towers) and destroy map objects.
- Enemies navigate the map using flow field pathfinding, with behavior driven by their internal state machine.
- Enemies can toggle or navigate around level features like bridges and doors (but cannot destroy them); the player can also manipulate these features.
- Game mechanics (enemy spawning, wave systems, UI) are implemented later.

**Out of Scope (Mechanics Deferred):**
- Player movement, aiming, and shooting mechanics
- Tower placement, tower behavior, and tower damage systems
- Wave/spawn systems and enemy spawning logic
- UI (HUD, health bars, menus, pause systems)
- Audio and visual effects

## Architecture

### 1. Flow Field System (DataGrid2D)

The **flow field** is the core pathfinding mechanism. It pre-computes distance information for every cell on the map, allowing enemies to quickly determine their direction and trigger behavior changes based on proximity to goals.

#### Key Components

**DataGrid2D** (`scripts/datagrid/datagrid2d.gd`)
- Merges all TileMapLayers into a unified grid with consistent coordinate space
- Auto-initializes `tilemap_layers` from TileMapLayer children of `StaticLayers`
- Auto-initializes `map_exits` from `MapExit` nodes under `StaticObjects`
- Cell size is configurable (default: 32 pixels)
- Reads custom `type` field from each tile in the TileMapLayers
- Stores multiple float arrays for flow fields: `exit_distance_field`, `player_distance_field`, and `flow_field_buffer`

**FlowField** (`scripts/datagrid/flowfield.gd`)
- Contains the flow-field calculation logic separate from `DataGrid2D`
- Writes new flow-field results into `flow_field_buffer`
- The intended algorithm is the fast sweep method with an eikonal solver
- For now, this file provides the scaffolding and buffer initialization only

**DataGrid2DCell** (`scripts/datagrid/datagrid2dcell.gd`)
- Stores static per-cell grid data such as walkability
- Flow-field distances are stored in float arrays on `DataGrid2D`, not on each cell

#### Distance Metrics & State-Driven Behavior

Enemies use flow fields to estimate distances to exits and to players or player-controlled NPCs. The flow field concept is simple:

1. **Each state has a primary field:** An enemy in `idle` state can follow `player_distance_field`; in `escape` state, it can follow `exit_distance_field`.
2. **Movement follows decreasing distance:** The enemy moves to adjacent cells where the active metric value is smaller (moving "downhill" on the field).
3. **State transitions trigger on field changes:** When the player-distance value drops below a threshold (e.g., 5 cells), the state machine transitions from `idle` to `chase_player`.
4. **Result:** Dynamic, context-aware behavior emerges without expensive per-enemy pathfinding calculations.

**Example Flow:**
- Enemy in `idle` state reads `player_distance_field` for its current cell
- If distance < 5 cells, state transitions to `chase_player`
- Enemy now follows `player_distance_field` until distance > 10 or other conditions trigger a different state

#### Updating the Flow Field

Whenever exits, players, player-controlled NPCs, or topology-relevant objects change, the relevant flow field must be recomputed:
- A new field is calculated on `flow_field_buffer`
- When the calculation finishes, `flow_field_buffer` is swapped with the active target field
- The old target field becomes the next reusable buffer
- The intended implementation uses the fast sweep method and eikonal solver in `flowfield.gd`
- All enemies automatically benefit from the updated field after the swap

### 2. State Machine (Enemy AI)

Enemies use a hierarchical state machine where each state is a child node of a StateMachine node.

#### State Base Class (State.gd)

All states inherit from `State`:

```gdscript
class_name State
extends Node

signal transitioned  # Emitted when state wants to change: transitioned.emit(new_state_name)

func enter():
    # Called when entering this state
    pass

func exit():
    # Called when leaving this state
    pass

func update(delta):
    # Called once per frame; for non-physics logic
    pass

func physics_update(delta):
    # Called once per physics frame; for movement, velocity, etc.
    pass
```

#### StateMachine Coordinator (state_machine.gd)

The StateMachine node:
1. Holds all state child nodes
2. Listens to each state's `transitioned` signal
3. Calls `exit()` on the current state, then `enter()` on the new state
4. Delegates `update()` and `physics_update()` calls to the active state each frame

#### Enemy State Example: enemy_idle_state.gd

```gdscript
extends State
class_name EnemyIdleState

@onready var enemy = get_parent().get_parent()  # Parent is StateMachine, grandparent is Enemy

func physics_update(delta):
    var flow_field_cell = enemy.get_flow_field_cell()
    
    # Check distance to player
    if flow_field_cell.distance_to_nearest_player < 5:
        transitioned.emit("chase_player")
        return
    
    # No movement in idle state
    enemy.velocity = Vector2.ZERO
```

### 3. TileMap Layer Structure

#### Layer Setup

Levels are constructed using **TileMapLayers** (Godot 4.x). Each layer represents a logical group of tiles (e.g., floor, walls, decorations).

#### Custom Data: "type" Field

Each tile in a TileMapLayer can have a custom data field called `type` (string). This field stores the tile's classification:

**Supported Types:**
- `"wall"` — Solid obstacle; blocks movement and projectiles
- `"walkable"` — Standard movement cell
- `"flyable"` — Cells passable by flying enemies (if implemented)
- Additional types can be defined per project (e.g., `"lava"`, `"water"`, `"spike"`, etc.)

**Usage in Code:**
```gdscript
var tile_data = tilemap_layer.get_cell_tile_data(cell_position)
var tile_type = tile_data.get_custom_data("type")  # Returns the "type" string
```

#### Initialization in DataGrid2D

When DataGrid2D initializes:
1. It auto-initializes `tilemap_layers` from TileMapLayer children of `StaticLayers`
2. It auto-initializes `map_exits` from `MapExit` nodes under `StaticObjects`
3. It merges the used rectangles of all TileMapLayers into a single `grid_rect`
4. For each layer, it retrieves all used cells and their custom "type" data
5. It adjusts cell positions relative to the unified grid origin (`grid_rect.position`)
6. It initializes the float-array flow fields and shared buffer based on the grid size
7. The grid is now ready for flow-field calculations and buffer-swapped updates

---

### 4. Interactive Level Items

Level features like **bridges** and **doors** can be toggled by the player:
- These are separate entities (CharacterBody2D, StaticBody2D, or Area2D nodes)
- They can be destroyed or disabled by the player
- When their state changes, the flow field should be recomputed (if they affect pathing)
- Enemies interact with them through collision physics or TileMap type updates

---

### 5. Destructible Objects

**Destructible objects** include:
- Player-placed towers (created at runtime)
- Map objects that can be damaged/destroyed (e.g., crates, pillars, enemy spawn points)

When a destructible object is placed or removed:
1. Update the flow field's `distance_to_nearest_destructible_object` metric
2. All enemies can immediately adjust behavior based on the new field

## Code Style & Conventions

### Variable Naming
- **Use full, descriptive names** — clarity over brevity
- ✅ `cell_position`, `distance_to_nearest_player`, `velocity_magnitude`
- ❌ `cell_pos`, `dist_player`, `vel_mag`

### Method Naming
- Local helper methods must not start with an underscore.
- Use full descriptive names, for example `initialize_tilemap_layers_from_static_layers` and `initialize_map_exits_from_static_objects`.
- Exception: Godot lifecycle callbacks keep their engine-defined names (for example `_ready`, `_process`, `_physics_process`, `_draw`).

### File Organization
```
scripts/
├── state_machine/
│   ├── state.gd                      # Base state class
│   └── state_machine.gd              # StateMachine coordinator
├── characters/
│   ├── player/
│   │   └── player.gd
│   └── enemy/
│       ├── enemy.gd                  # Enemy base class
│       └── states/
│           ├── enemy_idle_state.gd
│           ├── enemy_chase_player_state.gd
│           └── ... (other states)
├── datagrid/
│   ├── datagrid2d.gd                 # Flow field grid
│   └── datagrid2dcell.gd             # Single cell data
│   └── flowfield.gd                  # Flow field calculation logic
└── camera/
    └── camera.gd
```

### Class Naming
- State subclasses: `EnemyIdleState`, `EnemyChasePlayerState`, etc.
- Grid classes: `DataGrid2D`, `DataGrid2DCell`
- Use `class_name` for all custom classes

### Comments & Documentation
- Explain *why* logic exists, not just what it does
- Document flow field dependencies ("recompute flow field when X changes")
- Mark performance-critical sections with `# Performance: ...`

## Design Patterns

### State Machine Pattern
- Each state is a self-contained node
- States emit `transitioned` signal with the target state name
- StateMachine handles the state lifecycle (enter/exit)
- Prevents state conflicts through centralized coordination

### Lazy Distance Computation
- Cache flow-field values in float arrays on `DataGrid2D`
- Recompute into a shared buffer, then swap the buffer with the active field
- Only recompute when level topology or relevant source positions change
- Enemies read cached values every frame (O(1) lookup)

### Separation of Concerns
- **Flow field** handles "where should I go?" (distance metric)
- **State machine** handles "what should I do?" (behavior logic)
- **Enemy movement** is driven by state, not field directly

## Development Checklist

When implementing features:

1. **Flow Field Changes**
   - [ ] Update DataGrid2DCell to include/remove a distance metric if needed
   - [ ] Implement recomputation logic when cells change
   - [ ] Test that all enemies react appropriately to field updates

2. **New Enemy State**
   - [ ] Create `enemy_[state_name]_state.gd` inheriting from `State`
   - [ ] Implement `enter()`, `exit()`, `update()`, `physics_update()`
   - [ ] Emit `transitioned` signal with next state name
   - [ ] Register state in enemy's StateMachine as a child node

3. **Level Design**
   - [ ] Assign `type` field to all tiles in TileMapLayers
   - [ ] Verify walls block pathfinding (reflected in distance calculations)
   - [ ] Place exits and destructible objects, then recompute flow field

4. **Debugging**
   - [ ] Visualize flow field grid (overlay distance values or direction arrows)
   - [ ] Log state transitions to verify state machine correctness
   - [ ] Monitor distance metrics for each enemy in the debugger
